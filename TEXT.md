Введение
Работа заключается в создании frontend-части веб-приложения и сопутствующей инфраструктуры пользовательского интерфейса. Ключевым результатом стала разработка собственной библиотеки компонентов EOS UI Kit, которая легла в основу клиентского приложения.
Цель работы:
Разработка библиотеки компонентов EOS UI Kit и frontend-части веб-приложения Estate Operations System (EOS) для управления многоквартирным жилым комплексом с интеграцией мессенджера Telegram.
Задачи работы:
1.	Провести анализ существующих подходов к созданию UI-библиотек для Vue 3 и frontend-приложений в сфере ЖКХ.
2.	Разработать библиотеку компонентов EOS UI Kit: систему дизайн-токенов на CSS Variables, автоматическую генерацию иконок, набор базовых компонентов (включая таблицу с сортировкой и пагинацией), а также настроить публикацию npm-пакета.
3.	Обеспечить документирование библиотеки и визуальное регрессионное тестирование.
4.	Спроектировать архитектуру frontend-приложения.
5.	Реализовать аутентификацию пользователей, управление JWT-токенами и защиту маршрутов с разграничением доступа по ролям.
6.	Разработать API-клиент с автоматической генерацией TypeScript-типов на основе OpenAPI-схемы backend.
7.	Создать пользовательские интерфейсы для управления заявками и администрирования пользователей.
8.	Настроить end-to-end тестирование ключевых сценариев приложения и контроль качества кода.
9.	Выполнить production-сборку и развёртывание frontend-приложения на хостинге.
Глава 1. Разработка библиотеки компонентов
1.1 Анализ предметной области
В современных frontend-приложениях интерфейс состоит из большого количества повторяющихся элементов. При отсутствии централизованной библиотеки компонентов разработчики вынуждены повторно реализовывать одинаковые элементы интерфейса, что приводит к дублированию кода, усложнению сопровождения и нарушению единообразия UI.
Для решения данной проблемы используются дизайн-системы и библиотеки компонентов. Дизайн-система объединяет визуальные правила, набор компонентов и принципы проектирования интерфейсов, а её практическая реализация обычно включает дизайн-макеты в Figma и программную реализацию в виде npm-пакета.
Для экосистемы Vue существует ряд готовых библиотек компонентов, среди которых Vuetify, Element Plus, PrimeVue и Naive UI. Данные решения предоставляют широкий набор готовых компонентов и средств стилизации, однако требуют существенных усилий по переопределению стилей для соответствия уникальной дизайн-системе проекта, что иногда приводит к усложнению сборки или конфликтам стилей.
В связи с этим актуальной задачей является разработка собственной библиотеки компонентов, обеспечивающей гибкость настройки, единообразие интерфейсов и удобство дальнейшего сопровождения приложений.
1.2 Постановка требований к библиотеке компонентов
1.2.1 Назначение библиотеки
Разрабатываемая библиотека предназначена для создания и повторного использования UI-компонентов в приложениях на Vue 3.
Основу библиотеки составляют независимые компоненты и система дизайн-токенов, обеспечивающая настройку визуального оформления.
1.2.2 Функциональные требования
Библиотека должна предоставлять набор независимых UI-компонентов, пригодных для использования в различных Vue-проектах без существенных доработок.
В состав библиотеки входят кнопки, поля ввода и текстовые области, выпадающие списки, карточки, подальные окна, таблицы, индикаторы загрузки, теги.
Компоненты библиотеки должны поддерживать:
•	настройку через props и slots; 
•	различные визуальные состояния; 
•	обработку пользовательских событий; 
•	поддержку дизайн токенов и адаптивного отображения.

1.3 Выбор технологий и инструментов разработки
При разработке библиотеки компонентов был выбран технологический стек, обеспечивающий высокую скорость разработки, удобство сопровождения, масштабируемость и возможность повторного использования компонентов в различных проектах. Основу библиотеки составляют фреймворк Vue 3, язык TypeScript, сборщик Vite и система стилизации на базе SCSS и CSS Variables.
1.3.1 Выбор фреймворка и языка разработки
В качестве основного фреймворка был выбран Vue.js версии 3. Данный фреймворк предоставляет компонентный подход к разработке интерфейсов и обеспечивает удобные механизмы повторного использования UI-элементов.
Для реализации компонентов используется Composition API, позволяющий структурировать логику компонентов, улучшать переиспользуемость кода и упрощать сопровождение библиотеки.
В качестве языка разработки был выбран TypeScript. Использование статической типизации позволяет обнаруживать ошибки на этапе разработки, обеспечивает автодополнение в средах разработки и повышает удобство использования библиотеки в сторонних проектах.
1.3.2 Инструменты сборки и разработки
Для сборки библиотеки используется Vite. Основными причинами выбора являются высокая скорость запуска среды разработки, быстрая сборка проекта и поддержка современных форматов модулей.
В процессе разработки также использовался инструмент yalc, предназначенный для локального подключения npm-пакетов без публикации в удалённый реестр. Это позволило тестировать библиотеку в реальных проектах на этапе разработки.
Для обеспечения единообразия стиля кода применяются инструменты:
•	ESLint - статический анализ JavaScript и TypeScript-кода; 
•	Prettier - автоматическое форматирование кода; 
•	Husky - автоматический запуск проверок перед созданием коммитов.
1.3.3 Средства стилизации
Система стилизации библиотеки построена на основе SCSS, scoped styles, методологии BEM и CSS Variables. Синхронизация между Figma и кодом выполнялась вручную на основе согласованной системы именования, что позволило избежать внедрения дополнительных инструментов синхронизации
Использование scoped styles позволяет изолировать стили компонентов и избежать конфликтов с глобальными стилями проекта-потребителя. Методология BEM обеспечивает единообразную структуру CSS-классов и упрощает сопровождение компонентов.
Такой подход позволяет изменять внешний вид библиотеки без модификации кода компонентов и обеспечивает возможность адаптации библиотеки под различные дизайн-системы.
1.3.4 Инструменты тестирования и документации
Для документирования и изолированной разработки компонентов используется Storybook. Данный инструмент позволяет разрабатывать компоненты независимо от основного приложения, демонстрировать различные состояния компонентов и формировать интерактивную документацию.
Тестирование компонентов выполняется с использованием Playwright. Библиотека применяется для визуального тестирования компонентов в реальных браузерах и проверки корректности отображения интерфейсов.
Дополнительно используется Storycap, предназначенный для автоматического создания снимков компонентов в Storybook. Это позволяет реализовать visual regression testing и обнаруживать непреднамеренные изменения интерфейса при развитии библиотеки.
1.4 Проектирование библиотеки компонентов
1.4.1 Архитектура библиотеки
При проектировании архитектуры библиотеки компонентов особое внимание уделялось масштабируемости, удобству сопровождения и возможности повторного использования компонентов в различных проектах. Библиотека реализована как независимый npm-пакет EOS UI Kit, предназначенный для подключения в приложения на Vue 3.
Основой архитектуры является модульный подход, при котором каждый компонент представляет собой независимую единицу интерфейса со своей логикой, стилями и типизацией. Такой подход обеспечивает изоляцию компонентов, упрощает их сопровождение и позволяет расширять библиотеку без изменения существующей функциональности.
 
Рисунок 1. Структура библиотеки
Для каждого компонента внутри директории components создаётся отдельная папка, содержащая Vue-компонент, стили, типы и вспомогательные файлы. Такая организация упрощает навигацию по проекту и обеспечивает централизованное хранение связанных сущностей.
В качестве точки входа библиотеки используется файл index.ts, реализующий централизованный экспорт компонентов и глобальных стилей. Подобный подход (barrel exports) обеспечивает единую точку доступа к библиотеке и упрощает импорт компонентов в проектах-потребителях.
 
Рисунок 2. Пример импорта компонентов и типов
Библиотека поставляется в формате ESM, что позволяет использовать механизм tree shaking. Благодаря этому в итоговую сборку приложения включаются только реально используемые компоненты, что уменьшает размер итогового bundle.
Глобальные стили библиотеки организованы отдельно от компонентов, что обеспечивает централизованное управление визуальным оформлением. Помимо UI-компонентов библиотека экспортирует TypeScript-типы, SVG-иконки и вспомогательные утилиты, используемые при разработке интерфейсов.
Архитектура библиотеки ориентирована на независимость компонентов, минимизацию связности между модулями и возможность дальнейшего масштабирования системы без существенного изменения существующей структуры проекта.
1.4.2 Проектирование дизайн-системы
Проектирование дизайн-системы выполнялось в среде Figma и являлось основой для дальнейшей frontend-разработки библиотеки компонентов. Основной задачей дизайн-системы стало создание единого набора визуальных правил и интерфейсных элементов, обеспечивающих согласованность пользовательского интерфейса и возможность повторного использования компонентов.
Структура дизайн-системы организована в виде отдельных страниц, разделённых по функциональному назначению:
•	Tokens - дизайн-токены;
•	Icons - набор SVG-иконок;
•	Components - пользовательские компоненты и их состояния.
В основе дизайн-системы лежит система дизайн-токенов, описывающая основные визуальные параметры интерфейса. В рамках проекта были разработаны токенов:
•	цветовая палитра;
•	типографика;
•	отступы и скругления;
•	тени.
В Figma токены организованы в иерархическом формате (например, primary/50, spacing/m, radius/l). Для обеспечения согласованности между макетами и кодом была разработана единая система именования, преобразующая данные форматы в CSS-переменные вида --eos-color-primary-50, --eos-spacing-m, --eos-radius-l.
 
Рисунок 3. Переменные отступов и скруглений
Детальное описание реализации дизайн-токенов в кодовой базе библиотеки, включая примеры CSS-переменных и типографических классов, приведено в разделе 1.5.1.
Цветовая система библиотеки построена на основе набора primary-цветов и semantic-токенов для отображения различных состояний интерфейса. Для каждого цвета предусмотрено несколько уровней насыщенности, используемых для hover-, active- и disabled-состояний компонентов.
 
Рисунок 4. Переменные цветов
Типографическая система включает набор размеров текста, межстрочных интервалов и начертаний шрифтов, применяемых во всех компонентах библиотеки. Для формирования адаптивных интерфейсов также была разработана система spacing-токенов, определяющая внутренние и внешние отступы элементов.
 
Рисунок 5. Переменные типографики
Проектирование компонентов выполнялось с использованием возможностей Figma Component System. Все компоненты создавались на основе ранее определённых дизайн-токенов, что обеспечило визуальную согласованность интерфейса.
Для организации различных модификаций компонентов использовалась система variants. Внутри одного компонента объединялись:
•	размеры;
•	варианты оформления;
•	визуальные состояния.
Для интерактивных элементов были реализованы состояния default, hover, active, disabled, error и loading.
Component Properties использовались для настройки параметров компонентов без создания большого количества отдельных элементов. Данный механизм позволил управлять текстом, размерами, иконками и отображением отдельных частей компонента.
Для автоматического управления размерами и внутренними отступами компонентов применялся Auto Layout. Использование Auto Layout обеспечило:
•	адаптивное изменение размеров компонентов;
•	автоматическое выравнивание элементов;
•	упрощение сопровождения макетов;
•	единообразие структуры компонентов.
Interactive Components использовались для моделирования поведения интерфейсных элементов и визуализации пользовательских сценариев ещё до этапа frontend-разработки.
1.4.3 Проектирование системы иконок
Система иконок является отдельной частью дизайн-системы и предназначена для обеспечения визуальной согласованности пользовательского интерфейса. В качестве основы был выбран набор Material Design, отличающийся единым визуальным стилем, хорошей читаемостью и совместимостью с современными web-интерфейсами.
Все иконки библиотеки используют следующие параметры:
•	формат SVG;
•	размер 24×24 пикселя;
•	стиль отображения filled.
Использование SVG обеспечивает масштабируемость иконок без потери качества, а также возможность изменения цвета и размеров средствами CSS.
Для хранения иконок в дизайн-системе была создана отдельная страница Icons в Figma. Централизованная организация иконок упростила их повторное использование и обеспечила единообразие интерфейса.
Некоторые иконки были адаптированы под требования библиотеки компонентов. Например, отдельные элементы навигации и выпадающих списков были модифицированы для соответствия общей визуальной системе.
1.4.4 Принципы проектирования компонентов
При разработке библиотеки компонентов были сформулированы принципы проектирования, обеспечивающие универсальность, независимость и возможность повторного использования компонентов в различных проектах.
Одним из основных принципов является независимость компонентов. Каждый компонент библиотеки представляет собой изолированную единицу интерфейса и не содержит зависимостей от конкретного проекта, API или глобального состояния приложения. Все необходимые данные передаются через props, а взаимодействие с внешней средой осуществляется посредством событий.
При проектировании компонентов соблюдался принцип единственной ответственности (Single Responsibility Principle). Каждый компонент выполняет одну конкретную задачу и отвечает только за отображение и взаимодействие с пользователем

Глобальные стили библиотеки разделены на несколько файлов:
•	tokens.css - дизайн-токены; 
•	reset.css - нормализация браузерных стилей; 
•	typography.css - базовая типографика и utility-классы.

•	вложенность стилей; 
•	разделение стилей по компонентам; 
•	использование переменных и миксинов; 
•	более удобную организацию CSS-кода. 
Каждый компонент содержит собственные scoped-стили, изолированные от остального приложения. Это предотвращает конфликты CSS-классов и исключает влияние компонентов библиотеки на глобальные стили проекта-потребителя.

1.5.3 Реализация системы иконок
Для обеспечения единообразного визуального оформления интерфейса в библиотеке была реализована отдельная система SVG-иконок. Все иконки представлены в виде Vue-компонентов, что обеспечивает их удобное повторное использование внутри UI-компонентов библиотеки.
Для автоматизации работы с иконками был реализован pipeline генерации компонентов. Процесс обработки включает:
1.	чтение SVG-файлов из директории icons/;
2.	оптимизацию SVG через SVGO;
3.	удаление лишних атрибутов;
4.	замену фиксированных цветов на currentColor;
5.	генерацию Vue-компонентов;
6.	автоматическое создание экспортов.
Использование currentColor позволяет наследовать цвет иконок от родительского элемента и обеспечивает их корректную интеграцию в различные компоненты библиотеки.
Названия SVG-файлов автоматически преобразуются в PascalCase с добавлением суффикса Icon. Например, файл chevron-down.svg преобразуется в компонент EosChevronDownIcon.vue. Сгенерированные компоненты экспортируются централизованно и могут использоваться как обычные Vue-компоненты внутри приложения.
Автоматизация системы иконок позволила упростить сопровождение библиотеки, обеспечить единообразие интерфейса и исключить необходимость ручного обновления компонентов при изменении SVG-файлов.
1.5.4 Реализация базовых компонентов
Основу библиотеки составляют базовые пользовательские интерфейсные компоненты, предназначенные для построения различных типов интерфейсов. Все компоненты реализованы с использованием Vue 3 Composition API и TypeScript. При разработке соблюдались принципы, описанные в разделе 1.4.4.
Для большинства компонентов (Button, Input, Select, Modal, Card, Tabs, Tag, Loader) реализована стандартная функциональность: настройка через props, использование слотов для кастомизации содержимого, поддержка различных визуальных состояний (default, hover, active, disabled, loading) и обработка пользовательских событий.
Наиболее сложным компонентом библиотеки является Table. При его разработке особое внимание уделялось универсальности структуры данных, типобезопасности и поддержке расширенной функциональности, включающей сортировку и пагинацию с кнопкой "Показать ещё".

В отличие от типичных табличных компонентов, использующих плоский массив объектов, в данной реализации применяется гибкая структура, разделяющая столбцы, ячейки и строки.
Компонент поддерживает сортировку: клик по заголовку столбца генерирует событие sortChange с идентификатором столбца и направлением сортировки. Для работы с большими объёмами данных реализована пагинация с кнопкой "Показать ещё": при включении свойства paginationMode="button" под таблицей отображается кнопка, по нажатию которой генерируется событие loadMore. Управление пагинацией осуществляется через параметры pageSize (размер загружаемой порции), totalRows (общее количество строк для отображения прогресса), loadButtonText (настраиваемый текст кнопки), loading (блокировка повторных вызовов во время загрузки) и hasMore (скрытие кнопки при отсутствии данных). Дополнительно при включении свойства clickable клик по строке генерирует событие rowClick.

Для документирования библиотеки компонентов использовался Storybook. •	визуальных состояний;
•	интерактивных сценариев использования.

•	варианты оформления (primary, secondary, tertiary);

1.6.2 Тестирование компонентов
Для обеспечения стабильности пользовательского интерфейса и обнаружения непреднамеренных изменений внешнего вида компонентов в библиотеке было реализовано визуальное тестирование (visual regression testing). Тестирование выполнялось с использованием Playwright, Storybook и Storycap.
Основой процесса тестирования являлся Storybook, содержащий набор stories для всех компонентов библиотеки. Каждая story представляла отдельное состояние компонента, что позволяло выполнять тестирование компонентов в изолированной среде без зависимости от основного приложения.
Для компонентов библиотеки были реализованы screenshot tests, предназначенные для проверки корректности отображения интерфейса. Тестирование включало:
•	проверку отображения различных вариантов компонентов;
•	проверку визуальных состояний;
•	проверку корректности подключения и рендеринга компонентов;
•	обнаружение визуальных изменений после обновлений кодовой базы.
Для автоматического создания снимков компонентов использовался Storycap. Инструмент выполнял генерацию screenshot-снимков на основе stories, размещённых в Storybook. Для каждого компонента формировались baseline screenshots, используемые в качестве эталонных изображений при последующих проверках.
Процесс visual regression testing включал следующие этапы:
1.	запуск Storybook; 
2.	генерация screenshot-снимков компонентов; 
3.	сравнение новых изображений с baseline screenshots; 
4.	формирование отчёта о найденных различиях. 
При обнаружении изменений между изображениями формировался visual regression report, позволяющий определить наличие непреднамеренных изменений интерфейса. Полученные отчёты использовались при проверке изменений перед публикацией новых версий библиотеки.
Запуск тестирования выполнялся с помощью npm scripts, что обеспечивало возможность автоматизированной проверки библиотеки в процессе разработки. Визуальными тестами были покрыты все основные компоненты библиотеки.
Использование visual regression testing позволило повысить стабильность пользовательского интерфейса, упростить сопровождение библиотеки компонентов и снизить вероятность появления визуальных дефектов при развитии системы.

2.1 Анализ существующих решений
ГИС ЖКХ - наиболее распространённое решение в сфере цифровизации ЖКХ. Система позволяет передавать показания счётчиков, оплачивать услуги и направлять обращения. Однако frontend-платформы отличается сложной навигацией и перегруженным интерфейсом: даже базовые действия (подача обращения, поиск информации о доме) требуют множества переходов. Обязательная авторизация через Госуслуги дополнительно усложняет доступ.
Среди коммерческих решений выделяется Домиленд с современным интерфейсом и мобильным приложением. Но платформа ориентирована на крупные жилые комплексы, а избыток функциональных модулей повышает порог освоения.
Doma.ai использует автоматизацию и интеграцию с системами «умного дома», однако для небольших жилых объектов такая архитектура избыточна как функционально, так и с точки зрения сложности интерфейса.
Узкоспециализированные сервисы (например, Квартплата 24) решают отдельные задачи (оплата, начисления), но не обеспечивают полноценного управления домом.

2.3 Выбор технологий и инструментов разработки
При разработке frontend-приложения EOS выбран технологический стек, обеспечивающий высокую скорость разработки, масштабируемость, типобезопасность и удобство сопровождения.
2.3.1 Фреймворк и язык
В качестве основного фреймворка выбран Vue 3 благодаря современной реактивной системе, Composition API и удобной интеграции с TypeScript. Скорость разработки на Vue 3 выше по сравнению с React и Angular.
Nuxt выбран как meta-framework для Vue. Файловая маршрутизация ускоряет разработку и сокращает конфигурационный код, а встроенная поддержка middleware позволяет реализовать защиту маршрутов и авторизацию. Хотя текущая версия работает как SPA, использование Nuxt создаёт основу для возможного внедрения SSR в будущем.
TypeScript обеспечивает строгую типизацию на всех уровнях приложения: Vue-компоненты, composables, API-клиент, модели данных. Типы автоматически генерируются на основе OpenAPI-схемы backend, что гарантирует согласованность частей системы.
2.3.2 Пользовательский интерфейс
Для построения UI используется собственная библиотека компонентов EOS UI Kit, разработанная в рамках проекта. Отказ от готовых UI-библиотек (Vuetify, Element Plus, PrimeVue) обусловлен недостаточной гибкостью, несоответствием визуальному стилю и ограниченной функциональностью существующих решений. EOS UI Kit обеспечивает единый стиль интерфейсов, повторное использование компонентов, ускорение разработки новых страниц и унификацию поведения элементов. Компоненты применяются как в коде, так и в дизайн-макетах Figma, что гарантирует согласованность между проектированием и реализацией.
Стилизация frontend-приложения EOS построена на основе SCSS и scoped styles. Глобальные переменные (цвета, отступы, шрифты, брейкпоинты) наследуются из EOS UI Kit, что обеспечивает единый визуальный язык между библиотекой компонентов и приложением.
Для именования CSS-классов применяется методология BEM, обеспечивающая предсказуемую структуру и упрощающая сопровождение стилей. Общие стили вынесены в отдельный файл, а изоляция стилей компонентов реализована через scoped styles, что предотвращает конфликты с глобальными стилями.
CSS Variables для динамической смены темы в текущей версии не используются, но архитектура стилей допускает их внедрение в будущем без рефакторинга существующих компонентов.
2.3.3 Инструменты разработки
Для end-to-end тестирования frontend-приложения выбран Playwright. Данный инструмент обеспечивает кроссбраузерное тестирование, поддерживает ожидание появления элементов без искусственных задержек и предоставляет удобный отладчик. Альтернативные решения, такие как Cypress или Selenium, не рассматривались из-за более сложной настройки параллельного запуска и меньшей производительности при работе с динамическим контентом.
Для документирования API используется OpenAPI. Backend-приложение предоставляет OpenAPI-схему, на основе которой автоматически генерируются TypeScript-типы для frontend. Это позволяет поддерживать типобезопасность при вызовах API и обеспечивает актуальность документации - изменения в backend автоматически отражаются на frontend. Альтернативы, такие как ручное документирование или GraphQL, не рассматривались из-за дополнительных трудозатрат на поддержку.
Документация для разработчиков ведётся в формате README - это стандартный и наименее затратный способ поддержки актуальной информации о проекте.
Для сборки приложения используется Vite (быстрый запуск dev-сервера и горячая замена модулей). Для контроля качества кода применяются ESLint, Prettier, Husky и lint-staged.

Уровни приложения (пользовательский интерфейс, бизнес-логика, работа с данными) логически разделены, что упрощает сопровождение и расширение функциональности.
2.4.1 Общая архитектура приложения

•	api/ - API Client, OpenAPI-схема, сгенерированные типы, функции взаимодействия с backend;

2.5 Реализация аутентификации и авторизации
Система аутентификации и авторизации обеспечивает безопасный доступ пользователей к функциональности приложения и разграничение прав доступа в зависимости от роли. Реализованы авторизация по email и через Telegram, JWT-аутентификация с механизмом обновления токенов, а также выход из системы.
2.5.1 Способы авторизации
Авторизация по email является основным способом входа. Процесс включает ввод email, отправку кода подтверждения на почту и ввод полученного кода. Срок действия кода - 10 минут, при необходимости пользователь может запросить новый код. После успешного подтверждения backend возвращает access token, refresh token и информацию о роли пользователя. Отказ от хранения паролей повышает безопасность системы.
Авторизация через Telegram реализована с использованием виджета Telegram Login. Виджет передаёт на frontend данные пользователя и подпись для проверки подлинности. Полученные данные отправляются на backend, где выполняется проверка подписи, после чего создаётся либо загружается профиль пользователя.
2.5.2 Управление токенами
Access token автоматически добавляется в заголовок Authorization каждого запроса. Refresh token используется для обновления access token после истечения его срока действия. Оба токена сохраняются в localStorage, что позволяет сохранять пользовательскую сессию между перезагрузками страницы.
При получении ошибки 401 Unauthorized приложение автоматически отправляет refresh token на endpoint обновления токенов. После получения нового access token исходный запрос повторяется. Если refresh token оказывается недействительным, пользователь перенаправляется на страницу авторизации.
2.5.3 Выход из системы
При выходе из системы выполняются удаление токенов из localStorage, сброс состояния текущего пользователя, удаление сохранённой роли пользователя и перенаправление на главную страницу.
Особенность авторизации через Telegram: виджет Telegram Login использует глобальную сессию, общую для всех сайтов в рамках одного браузера. Выход из учётной записи Telegram на уровне приложения EOS технически невозможен - пользователь может завершить сессию только непосредственно в самом Telegram. Приложение EOS при этом очищает только свои локальные токены и данные пользователя.
2.5.4 Защита маршрутов и разграничение доступа
Защита маршрутов и проверка ролей реализована через middleware Nuxt (подробно описано в разделе 2.4.3). Дополнительно ограничения реализуются на уровне интерфейса: функции управления пользователями доступны только администраторам, жильцы имеют доступ только к собственным данным и заявкам, отдельные элементы интерфейса скрываются в зависимости от роли пользователя.
2.6 Работа с API и управление состоянием
Frontend-приложение взаимодействует с backend через REST API. Для организации работы с данными реализован API Client, инкапсулирующий HTTP-запросы, обработку ошибок и управление авторизацией. Управление состоянием построено на реактивной системе Vue и Composition API.
2.6.1 API Client и типизация
API Client централизованно выполняет запросы к backend, добавляет токены авторизации и обрабатывает ошибки. Реализация выполнена на нативном fetch без использования сторонних HTTP-клиентов. Применение отдельного клиента позволяет избежать дублирования логики запросов внутри компонентов и composables.
Для временных сетевых ошибок реализован механизм автоматического повторения запросов (retry logic). При сетевой ошибке выполняется до 3 попыток с задержкой в 1 секунду между ними. Ошибки авторизации (401) обрабатываются отдельно: инициируется обновление access token. Остальные ошибки (серверные, клиентские) обрабатываются единым образом с отображением пользовательского сообщения.
Для обеспечения типобезопасности используется автоматическая генерация TypeScript-типов на основе OpenAPI-схемы backend. Генерация выполняется командой npm run gen:api. В результате формируются типы моделей данных, endpoints, параметров запросов и ответов сервера. Такой подход обеспечивает согласованность frontend и backend, автоматическое обновление типов при изменении API и улучшенную поддержку в IDE.
2.6.2 Управление состоянием и composables
Для повторного использования бизнес-логики применяются composables. Основной из них - useAuth (описан в разделе 2.5) - отвечает за авторизацию, хранение пользователя, управление токенами, загрузку профиля и выход из системы. Использование composables позволяет изолировать бизнес-логику от компонентов и уменьшить дублирование кода.
Для управления локальным состоянием компонентов используются реактивные API Vue: ref, reactive, computed, watch. Реактивное состояние применяется для хранения данных форм, отображения списков, управления загрузкой, отображения ошибок, фильтрации и поиска данных.

Для обеспечения стабильности работы frontend-приложения и поддержания высокого качества кода в проекте используются инструменты статического анализа, автоматического форматирования, автоматизированного тестирования и документирования.
2.8.1 Инструменты контроля качества кода
Для статического анализа JavaScript и TypeScript кода используется ESLint, обеспечивающий контроль качества, поиск потенциальных ошибок и соблюдение единого стиля разработки. Для автоматического форматирования кода применяется Prettier, гарантирующий единообразное оформление всех файлов проекта.
Автоматические проверки перед коммитом настроены с использованием Husky и lint-staged. Перед выполнением commit запускаются ESLint и проверка форматирования, что предотвращает попадание некорректного кода в репозиторий. 
2.8.2 Тестирование приложения
End-to-end тестирование выполняется с использованием Playwright. Автотесты покрывают ключевые пользовательские сценарии: авторизацию (email и Telegram), создание и изменение заявок, изменение статусов (для администратора), редактирование профиля и выход из системы.
Ручное тестирование дополняет автоматизированное и ориентировано на проверку адаптивности интерфейсов (мобильные устройства, планшеты, desktop), корректности отображения данных и проверку сложных сценариев взаимодействия с backend (недоступность сервиса, ошибки сети, истечение токена).
Проверка API-взаимодействия включает корректность HTTP-запросов, обработку ответов сервера, работу JWT-аутентификации, обновление refresh token и обработку ошибок 401, 403, 500.
2.8.3 Документация проекта
Документация frontend-приложения ведётся в файле README корневого репозитория и включает:
•	инструкцию по установке и запуску проекта;
•	описание структуры директорий;
•	перечень используемых команд (сборка, линтинг, тестирование, генерация типов);
•	указания по работе с API-клиентом.
Документация по API-запросам обеспечивается автоматически: типы данных и схемы запросов генерируются на основе OpenAPI-спецификации backend (описано в разделе 2.6.1), что служит актуальной документацией для взаимодействия с серверной частью.
2.9 Развёртывание frontend-приложения
2.9.1 Сборка production-версии
Для подготовки frontend-приложения к эксплуатации используется production-сборка Nuxt, выполняемая командой npm run build. В процессе сборки компилируется TypeScript-код, оптимизируются ресурсы и формируется итоговый набор статических файлов для запуска приложения.
Nuxt и Vite обеспечивают встроенную оптимизацию: минимизацию JavaScript и CSS, удаление неиспользуемого кода, code splitting и lazy loading страниц. Использование production-сборки позволяет уменьшить размер итогового бандла и повысить скорость загрузки приложения в браузере пользователя.
2.9.2 Развёртывание на хостинге
Приложение развёртывается в виде статических файлов на российском хостинге. Процесс развёртывания включает следующие шаги:
1.	выполнение production-сборки (npm run build);
2.	загрузка сгенерированных файлов из директории .output/public/ на сервер хостинга;
3.	Взаимодействие frontend-приложения с backend осуществляется через HTTP API (подробно описано в разделе 2.6). Базовый адрес backend-сервера задаётся на этапе сборки через переменные окружения, которые используются только в процессе сборки и не требуют наличия .env файла на сервере.
2.10 Перспективы развития
Архитектура frontend-приложения спроектирована с учётом возможности дальнейшего расширения функциональности системы.
Использование Nuxt позволяет в будущем реализовать поддержку серверного рендеринга (SSR). Несмотря на то что в текущей версии SSR не используется (приложение работает как SPA), выбранный стек позволяет внедрить SSR без полной переработки архитектуры. Это может быть полезно при создании публичных страниц, индексируемых поисковыми системами.
Компонентная архитектура и использование Composition API позволяют добавлять новые модули без существенного изменения существующего кода. В дальнейшем возможно:
•	добавление новых страниц и разделов;
•	расширение функциональности заявок;
•	внедрение дополнительных сервисов;
•	подключение новых интеграций.
Система ролей пользователей может быть дополнена новыми типами, например, сотрудники управляющей компании, технические специалисты, операторы поддержки, модераторы системы.
В перспективе возможно развитие технологической базы:
•	внедрение централизованного глобального состояния через Pinia;
•	добавление unit-тестирования;
•	внедрение CI/CD процессов для автоматического развёртывания;
•	усиление системы аналитики и мониторинга.
Использование собственной библиотеки EOS UI Kit обеспечивает возможность дальнейшего развития интерфейсов приложения. При добавлении новых компонентов сохраняется единый визуальный стиль и согласованность пользовательского интерфейса.
 
Заключение
В результате выполнения дипломной работы была разработана frontend-часть веб-платформы, предназначенной для цифровизации процессов управления многоквартирным жилым комплексом. Также важной частью является создание собственной библиотеки пользовательских интерфейсных компонентов EOS UI Kit.
Был решены все поставленные задачи. Проведен анализ существующих популярных решений по данной теме с точки зрения эффективности frontend-части. Рассмотрены особенности современных систем для управления жилыми комплексами и обозначены их недостатки. На основании проведенного анализа были сформированы требования к системе и сделан упор на улучшение найденных проблем.
Важным итогом стала разработка EOS UI Kit - независимой библиотеки компонентов для Vue 3. Она была реализован в виде npm-пакета и выложена в открытый доступ на официальный сайт.  Данная библиотека включает в себя систему дизайн-токенов и SVG-иконок, набор базовых UI-компонентов и единый механизм стилизации. Модульная архитектура позволяет обеспечить надежность системы, переиспользуемых. 
Была подготовлена документация в Storybook. Каждый компонент библиотеки сопровождался набором stories, которые представляют собой набор имеющихся состояний и сценариев. Такой подход позволил детально прорабатывать поведение компонентов до их интеграции в основную часть frontend-приложения. 
На основании разработанной дизайн-системы было создано frontend-приложение, включающее в себя систему аутентификации по email и Telegram, ролевую модель пользователей, модуль управления заявками, API-клиент с автоматической генерацией TypeScript-типов и адаптивные пользовательские интерфейсы.
