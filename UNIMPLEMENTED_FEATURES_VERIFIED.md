# Нереализованный функционал frontend-приложения Estate Operations System

На основе анализа Главы 2 "Разработка приложения" диплома выявлены следующие элементы, которые описаны как требования или планируемые функции, но не реализованы в текущей версии приложения.

---

## 1. Требования, описанные в 2.2.2, но не реализованные

### 1.1 Real-time функциональность
- **Требование**: "отображение состояния заявок и данных в реальном времени" (2.2.2)
- **Реальность**: 
  - В 2.7.3 описано только "обновление данных после изменений" (ручное обновление)
  - Нет использования WebSocket или других механизмов real-time обновлений
  - Изменения доступны только после ручного обновления страницы

---

## 2. Функциональные возможности, указанные как перспективы развития в 2.10

### 2.1 Глобальное управление состоянием с Pinia
- Описано в 2.10: "внедрение централизованного глобального состояния через Pinia"
- Текущая реальность: используется локальное состояние composables (2.6.2)

### 2.2 Серверный рендеринг (SSR)
- Описано в 2.10: "Использование Nuxt позволяет в будущем реализовать поддержку серверного рендеринга (SSR)"
- Текущая реальность: приложение работает как SPA (Single Page Application)

### 2.3 Unit-тестирование
- Описано в 2.10: "добавление unit-тестирования"
- Текущая реальность: используется только end-to-end тестирование с Playwright (2.8.2)

### 2.4 CI/CD процессы
- Описано в 2.10: "внедрение CI/CD процессов для автоматического развёртывания"
- Текущая реальность: ручное развёртывание через загрузку файлов на хостинг (2.9.2)

### 2.5 Система аналитики и мониторинга
- Описано в 2.10: "усиление системы аналитики и мониторинга"
- Текущая реальность: аналитика и мониторинг не упоминаются в описании текущей реализации

### 2.6 Расширение системы ролей пользователей
- Описано в 2.10: "В перспективе возможно развитие технологической базы... Система ролей пользователей может быть дополнена новыми типами, например, сотрудники управляющей компании, технические специалисты, операторы поддержки, модераторы системы"
- Текущая реальность: реализованы три роли - жилец, администратор, юрист (2.4.3)

### 2.7 Расширение функциональности заявок
- Описано в 2.10: "расширение функциональности заявок"
- Текущая реальность: базовая функциональность (создание, просмотр, изменение статуса)

---

## 3. Упомянутые в тексте, но не описанные подробно функции

### 3.1 Комментарии к заявкам
- Упоминание: "Детальный просмотр заявки отображает полную информацию: описание, текущий статус, дату создания, комментарии и данные об авторе" (2.7.3)
- Неясность: не описано, как пользователи добавляют комментарии к заявкам, интерфейс для этого

### 3.2 Адаптивный дизайн для мобильных устройств
- Упоминание: "ручное тестирование... ориентировано на проверку адаптивности интерфейсов (мобильные устройства, планшеты, desktop)" (2.8.2)
- Неясность: детали реализации адаптивности не описаны, нет упоминания о мобильном приложении

---

## 4. Элементы, подготовленные к использованию, но не реализованные

### 4.1 Динамическая смена темы (Dark Mode)
- Подготовка: "CSS Variables подготовлены к внедрению динамической смены темы" (2.3.2)
- Текущая реальность: темы/переключение темы не реализовано

### 4.2 Retry-логика для сетевых ошибок
- Описано наличие: "Для временных сетевых ошибок реализован механизм автоматического повторения запросов (retry logic)" (2.6.1)
- **Примечание**: Это РЕАЛИЗОВАНО, просто упомянуто как уже готовое решение
